<task type="auto">
  <name>Task 1: Create Dockerfile and .dockerignore for Railway deployment</name>
  <files>backend/Dockerfile, backend/.dockerignore</files>
  <action>
Create backend/Dockerfile:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# System dependencies required by torchaudio (libsndfile1) and audio processing (ffmpeg)
# Both are required — missing libsndfile1 causes silent torchaudio failures
RUN apt-get update && apt-get install -y --no-install-recommends \
    libsndfile1 \
    ffmpeg \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
# Install Python dependencies — torch 2.3 download is ~800MB, cached after first build
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code AFTER requirements to maximize layer cache hits
COPY . .

# Single worker is mandatory: multiple workers load the model N times, exceeding Railway's 512MB RAM limit
# PORT env var is provided automatically by Railway
CMD ["sh", "-c", "uvicorn main:app --host 0.0.0.0 --port ${PORT:-8000} --workers 1"]
```

Key constraints:
- `--workers 1` is mandatory: with multi-worker, each worker loads Wav2Vec2 (~450MB) independently, causing OOM on Railway free tier (512MB limit)
- `libsndfile1` and `ffmpeg` are both required system packages — torchaudio's audio I/O fails silently without them
- Use `${PORT:-8000}` to respect Railway's PORT env injection (Railway sets PORT automatically)
- Do NOT add HEALTHCHECK directive — Railway uses the /health endpoint via HTTP probe

Create backend/.dockerignore:

```
__pycache__/
*.pyc
*.pyo
*.pyd
.env
.env.*
test_client.py
test_audio/
*.wav
*.mp3
.git
.gitignore
```

Excluding test_audio/ and *.wav keeps the image lean — test audio files (potentially large WAVs) must not be baked into the production image.

After creating both files, verify Docker build locally:
`docker build -t noscam-backend ./backend`

If Docker is not available locally, note this in the output and proceed to Railway deployment. Railway builds from the Dockerfile directly.

<task type="auto">
  <name>Task 2: Create railway.json and push to GitHub</name>
  <files>backend/railway.json</files>
  <action>
Create backend/railway.json to tell Railway how to build and start the service:

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "Dockerfile"
  },
  "deploy": {
    "startCommand": "uvicorn main:app --host 0.0.0.0 --port $PORT --workers 1",
    "healthcheckPath": "/health",
    "healthcheckTimeout": 120,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 3
  }
}
```

healthcheckTimeout of 120 seconds is intentional: Wav2Vec2 model download on first deploy can take 60-90 seconds on Railway's network. Railway will wait 120s for /health to return 200 before marking the deploy failed.

healthcheckPath "/health" causes Railway to poll GET /health — it will wait for the {"status": "ready"} response (model loaded) before routing traffic.

After creating railway.json, verify JSON syntax:
`python3 -c "import json; json.load(open('backend/railway.json')); print('valid JSON')"`

Then stage and push all backend files to GitHub so Railway can pick them up:
`git add backend/ && git commit -m "feat(01-backend): add FastAPI inference service with Docker" && git push`

Note: The Railway dashboard connection (New Project -> Deploy from GitHub -> set Root Directory to "backend") must be done by the user — see user_setup. Claude cannot authenticate to the Railway dashboard.
